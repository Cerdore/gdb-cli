### 6.2 修复的问题（2026-03-20）

修复了 3 层嵌套 Bug，最终 E2E 测试全部通过：

#### Bug 1: Socket 通信协议死锁
- **根因**: Client `sendall()` 后不关闭写端 → Server recv 循环等待 EOF → 双方互等 → 超时
- **修复**: `client.py` — 在 `sendall()` 后调用 `socket.shutdown(SHUT_WR)`，每次 call 后 close 重连

#### Bug 2: GDB 进程无法常驻
- **根因**: Launcher 用 `os.pipe()` 保持 GDB stdin，但 CLI 进程退出后 pipe write fd 关闭 → GDB 收到 EOF 退出
- **修复**: `launcher.py` — 用 named FIFO (O_RDWR) 替代 pipe，GDB 阻塞在 FIFO 读端，不依赖父进程

#### Bug 3: Session 元数据未持久化 gdb_pid
- **根因**: `_start_gdb_process()` 设置 `session.gdb_pid` 后未回写 `meta.json` → 后续读取 session 判定为 dead → 被清理
- **修复**: `launcher.py` — 在设置 gdb_pid 后调用 `_write_meta(session)`

#### Bug 4: `gdb.Value` 的 `str()` 从后台线程触发 GDB SEGFAULT
- **根因**: `val.type.is_unsigned()` 在 GDB 12.1 不存在 → `_format_int` 的 except 分支调用 `str(val)` → 触发 Guile pretty printer（非线程安全）→ SEGFAULT
- **修复**: `value_formatter.py` — 添加 `_safe_str()` 辅助函数（用 `format_string(raw=True)` 绕过 pretty printer），替换所有 `str(val)` 调用；`is_unsigned()` 改为 `hasattr` 保护

#### 架构决策：放弃 `gdb.post_event()`
- GDB 12.1 在非交互模式下不处理 `post_event` 队列
- 对于 core dump 分析（只读场景），后台线程直接 dispatch 是正确的务实选择
- 避免调用 `str(gdb.Value)` 即可规避大部分线程安全问题

### 8.5 关键发现：`gdb.execute()` 后台线程 SEGFAULT 根因分析

#### 8.5.1 GDB 官方文档的线程安全约束

根据 GDB 官方文档（"Basic Python" / "Threading in GDB"）：

> **GDB is not thread-safe.** If your Python program uses multiple threads, you must be careful to only call GDB-specific functions in the GDB thread.

> `gdb.post_event(event)` — This is the **only thread-safe way** to call into GDB from another thread.

即：**所有** GDB Python API（包括 `gdb.execute()`、`gdb.parse_and_eval()`、`gdb.selected_frame()` 等）从非主线程调用均属于 undefined behavior。

#### 8.5.2 实际测试：哪些 API 在后台线程真正 crash

在 `work@6.12.234.100` (GDB 9.2) 上对真实 DB core dump (6.1G binary, 779 线程) 做了逐项验证：

| API 调用 | 小程序 (1 线程) | DB core (779 线程) | 说明 |
|----------|:-:|:-:|------|
| `gdb.selected_inferior()` | ✅ | ✅ | 纯 Python 对象访问 |
| `inferior.threads()` | ✅ | ✅ | 读取线程列表 |
| `gdb.selected_thread()` | ✅ | ✅ | 读当前线程 |
| `thread.switch()` | ✅ | ✅ | 切换线程上下文 |
| `gdb.selected_frame()` | ✅ | ✅ | 读当前帧 |
| `gdb.newest_frame()` | ✅ | ✅ | 获取最新帧 |
| `frame.older()` | ✅ | ✅ | 遍历栈帧链 |
| `frame.name()` | ✅ | ✅ | 读函数名 |
| `frame.select()` | ✅ | ✅ | **Python API 选帧，安全** |
| `frame.block()` + `sym.value()` | ✅ | ✅ | **遍历局部变量，安全** |
| `gdb.parse_and_eval("1+1")` | ✅ | ✅ | 简单表达式求值 |
| `gdb.parse_and_eval("this")` | ✅ | ✅ | 复杂指针求值也安全（求值本身不 crash） |
| `str(gdb.Value)` 复杂对象 | ✅ | **❌ SEGFAULT** | 触发 pretty printer（Guile/Python），非线程安全 |
| `val.format_string(raw=True)` | ✅ | ✅ | **绕过 pretty printer，安全** |
| `gdb.execute("print 1+1")` | ✅ | **❌ SEGFAULT** | 即使最简单的命令也 crash |
| `gdb.execute("info threads")` | ✅ | **❌ SEGFAULT** | 同上 |
| `gdb.execute("frame 3")` | ✅ | **❌ SEGFAULT** | 同上 |

#### 8.5.3 两类 crash 的根因对比

**Crash 类型 A: `gdb.execute()` — CLI 解释器竞争**

```
后台线程 → gdb.execute("frame 3") → GDB CLI interpreter
  → 命令解析 → 输出重定向 → 上下文切换 → 全局状态修改
  → 与主线程的 batch-mode 事件循环竞争 → SEGFAULT
```

- `gdb.execute()` 走 GDB 的 **CLI command interpreter** 完整路径
- 涉及命令解析、输出缓冲区分配、pager 状态等大量全局状态
- 小程序碰巧能用（符号表简单，竞争窗口极小）
- 大型 DB binary (779 线程, 6.1G) **100% 复现**

**Crash 类型 B: `str(gdb.Value)` — Pretty Printer 竞争**

```
后台线程 → str(val) → Python __str__ → gdb valpy_str()
  → 查找 pretty printer → 触发 Guile SCM pretty printer
  → scm_gc_protect_object() → GC 竞争 → SEGFAULT
```

- `str(val)` 内部调用 `valpy_str()` → 自动查找 pretty printer
- DB 注册了 Guile pretty printer，Guile GC 不是线程安全的
- **修复**: 使用 `val.format_string(raw=True)` 绕过 pretty printer

#### 8.5.5 结论：后台线程安全的 GDB Python API 边界

在 core dump 只读场景下，后台线程可安全调用的 API：

| 安全 ✅ | 不安全 ❌ |
|---------|----------|
| `gdb.selected_inferior()` | `gdb.execute()` — 任何命令 |
| `inferior.threads()` | `str(gdb.Value)` — 触发 pretty printer |
| `thread.switch()` | |
| `gdb.newest_frame()` / `frame.older()` / `frame.select()` | |
| `frame.name()` / `frame.sal()` / `frame.block()` | |
| `gdb.parse_and_eval(expr)` | |
| `sym.value(frame)` | |
| `val.format_string(raw=True)` | |
| `int(val)` / `float(val)` | |

**核心规则**：
1. **永远不用 `gdb.execute()`** — 用对应的 Python API 替代
2. **永远不用 `str(gdb.Value)`** — 用 `val.format_string(raw=True)` 替代
