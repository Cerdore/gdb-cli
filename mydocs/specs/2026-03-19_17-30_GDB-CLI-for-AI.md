## 1. Requirements (Context)

- **Goal**: 构建一个"瘦客户端 CLI + GDB 内置 Python RPC Server"架构的 GDB 调试工具，使 Claude Code 能通过 Bash 工具完成有状态的 GDB 调试（core dump 分析 & live attach），输出 LLM 友好的结构化 JSON
- **In-Scope**:
  - Core dump 加载与交互式探索（符号表常驻内存，毫秒级响应）
  - Live attach 到运行中进程（non-stop 模式 + scheduler-locking）
  - 瘦 CLI 客户端（Unix Domain Socket 通信）
  - GDB 内置 Python RPC Server（gdb.post_event 线程安全）
  - 结构化 JSON 输出 + 自动截断/分页 + hint
  - 会话生命周期管理（心跳超时自动 detach + quit）
  - 命令安全白名单（attach 模式下限制危险操作）
  - 幂等性保证（同 PID 不重复 attach）
  - debuginfo/符号路径自动探测与提示
- **Out-of-Scope**:
  - 远程调试（target remote）
  - GUI 可视化
  - MCP Server 协议层（与 gdb-mcp 共存但不复用 MCP 传输）
  - 多 inferior 调试

## 2. Research Findings

### 2.1 核心矛盾
- CLI 每次调用是独立进程（无状态）；GDB 调试会话是强状态的（断点、帧选择、线程上下文）
- 解法：**GDB 内置 Python RPC Server** — 让 GDB 进程本身成为有状态的微服务，CLI 仅做 Socket 客户端

### 2.2 为什么选 GDB Python API 而不是 MI 协议
| 维度 | GDB/MI (pygdbmi) | GDB Python API (gdb module) |
|---|---|---|
| 数据格式 | MI 文本，需二次解析嵌套转义 | Python 原生对象，直接 json.dumps |
| 复杂结构体遍历 | 只能逐字段 -data-evaluate-expression | gdb.parse_and_eval() 递归遍历 |
| non-stop 支持 | pygdbmi 异步事件处理不完善 | gdb.events 原生支持 |
| TTY/分页问题 | 需处理 "Type return to continue" | Socket 通信，无 TTY |
| 安全性 | 外部进程，GDB 崩溃连带影响 | 内部 try-except，异常不崩 GDB |

### 2.3 数据库调试场景特殊需求
1. **符号加载**：生产环境 binary 通常 strip，debuginfo 在单独文件或 debuginfo rpm 中。CLI 启动时需自动探测并提示
2. **core 文件巨大**：几十~上百 GB，加载耗时数分钟 → 必须常驻，不能 batch
3. **跨机器调试**：core 传到其他机器后 so 路径不同 → 需支持 `set solib-absolute-prefix` / `set sysroot`
4. **多线程**：几百~上千线程，`info threads` 输出会撑爆 token → 强制截断 + 分页
5. **scheduler-locking on**：attach 活进程调试时，只让当前线程动，防止其他线程改变状态
6. **gdb 版本**：需要 gdb 9.0+，rockylinux/el9 需要 gdb 15+

### 2.6 关键技术细节

#### gdb.post_event() — 线程安全的唯一手段
```python
import threading, socket, json, queue
import gdb

class GDBRPCServer:
    def _accept_loop(self):
        """后台线程：只负责 I/O，绝不碰 GDB API"""
        while self.running:
            conn, _ = self.server_sock.accept()
            req = json.loads(conn.recv(MAX_REQUEST_SIZE))
            result_q = queue.Queue()

            def handler(req=req, q=result_q):
                try:
                    result = self._dispatch(req)
                    q.put({"ok": True, "data": result})
                except gdb.MemoryError as e:
                    q.put({"ok": False, "error": f"Cannot access memory: {e}"})
                except gdb.error as e:
                    q.put({"ok": False, "error": str(e)})
                except Exception as e:
                    q.put({"ok": False, "error": str(e)})

            gdb.post_event(handler)  # ★ 投递到 GDB 主事件循环

            resp = result_q.get(timeout=COMMAND_TIMEOUT)
            conn.sendall(json.dumps(resp).encode())
            conn.close()
```

#### 自动截断策略
- 截断时保留首尾 + 摘要，而非直接丢弃
- 每个返回值标注 `truncated: true/false` + `total_count` + `hint`
- 示例（threads 返回上千线程时）：
```json
{

    {"id": 1, "name": "main", "state": "stopped", "frame": "..."},
    {"id": 2, "name": "TNTL0", "state": "running", "frame": "..."},
    "... 998 more threads ...",
    {"id": 1000, "name": "WR_T999", "state": "blocked", "frame": "..."}
  ],
  "total_count": 1000,
  "truncated": true,
  "hint": "use 'gdb-cli threads --range 3-20' for specific threads"
}
```

#### 大对象分页（Buffer Pool / Lock Table 等）

### 2.7 安全机制

#### 命令白名单 (attach 模式)
| 类别 | 允许 | 限制/禁止 |
|---|---|---|
| 读取类 | bt, info threads, print, x, ptype, info locals | 无限制 |
| 帧/线程导航 | frame, thread, up, down | 无限制 |
| 执行控制 | continue, step, next, finish | 需确认（可配置） |
| 内存修改 | set variable | **默认禁止**，需 --allow-write |
| 函数调用 | call | **默认禁止**，需 --allow-call + 超时 |
| 进程信号 | signal | **禁止** |

#### 幂等性
- 同一 PID 同时只允许一个 session
- `gdb-cli attach --pid 1234` 若已存在 → 返回已有 session_id
- 防止 AI 重试导致重复 attach

#### 心跳超时
- 默认 10 分钟无操作 → 自动 detach + quit
- 可通过 `--timeout 30m` 配置
- 超时序列：`detach (graceful)` → `quit` → 3s 后 `SIGKILL`（兜底）
- 清理 socket 文件和 meta.json

### 2.8 权限与环境自检
- ptrace 权限检测：`/proc/sys/kernel/yama/ptrace_scope`
- GDB 版本检测：最低 9.0，推荐 15+（for rockylinux/el9）
- debuginfo 探测：`readelf -S binary | grep debug_info`，缺失则返回结构化建议
- 源码路径映射：支持 `--source-dir` 设置 `directory` 指令

## 3. Innovate: Options & Decision

### Option A: Python CLI（推荐）
- 实现：CLI 用 Python + Click/Typer，GDB Server 用 GDB 内嵌 Python
- Pros:
  - 开发速度最快，一种语言搞定
  - GDB Python API 文档和生态成熟
  - 可以共享类型定义和序列化逻辑
  - pip install 即可分发
- Cons:
  - CLI 启动速度比 Go 慢 50-100ms（对 AI 场景可接受）
  - 需要目标机器有 Python 3.8+

### Option B: Go CLI + Python GDB Server
- 实现：CLI 用 Go 编译为单 binary，Server 仍用 GDB 内嵌 Python
- Pros:
  - CLI 零依赖，单 binary 分发
  - 启动极快（<10ms）
- Cons:
  - 两种语言维护成本
  - JSON 协议层需双重定义
  - GDB Server 部分仍需 Python，分发时要同时带 .py 文件

### Decision
- **Selected: Option A (Python CLI)**
- **Why**:
  - 目标用户是数据库内核开发者，机器上必然有 Python
  - 开发周期短，一个人维护成本低
  - CLI 的 50ms 启动延迟 vs GDB 命令执行时间（100ms~数秒），完全可忽略
  - 统一语言便于 GDB Server 和 CLI 共享 JSON schema、truncation 逻辑等
